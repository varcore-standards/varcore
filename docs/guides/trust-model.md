# NonSudo Trust Model

> **VAR v1.0 — 2026-03-01**

This document describes the trust assumptions, verification layers, and threat model for
the NonSudo Verifiable Action Receipt (VAR) system.

---

## 1. Overview

NonSudo VAR produces a cryptographically signed, tamper-evident audit trail of every action
taken by an AI agent.  Each *receipt* is an Ed25519-signed JSON document.  A sequence of
receipts forms a *chain* where each entry commits to the hash of its predecessor, making
the ordering and completeness of the record verifiable after the fact.

The system is designed so that verification can be performed entirely offline against a
locally-cached public key.  No trust in the NonSudo service is required to verify receipts
you already hold.

---

## 2. Principals

| Principal | Role |
|-----------|------|
| **NonSudo Proxy** | Signs every receipt at the point of action using the operator's Ed25519 private key.  This is the sole signing authority. |
| **AI Agent** | Submits tool-call requests through the proxy.  Agents are untrusted; every request is policy-checked before a receipt is written. |
| **Workflow Initiator** | The human or system that starts a session.  Recorded in `workflow_manifest.initiator_id`. |
| **Policy Author** | The operator who writes the `nonsudo.yaml` policy file.  Policy is evaluated at signing time; its hash is committed in the manifest. |
| **Verifier** | Any party who later verifies the receipt chain (auditor, CI system, another service). |
| **TSA** | An optional RFC 3161 Time-Stamp Authority that provides external time-bound proof. |

## 3. Key Management

> For operational procedures (rotation, compromise response, offline verification), see
> [Key Management](key-management.md).  For data retention and storage policies, see
> [Data Handling](data-handling.md).

### 3.1 Signing Key

- A single **Ed25519** key pair is generated by the proxy on first run.
- The **private key** is stored at `~/.nonsudo/keys/<key_id>.key` as a hex string, with
  file mode **0600** (readable only by the owner).
- The **public JWK** is stored at `~/.nonsudo/keys/<key_id>.jwk`, file mode **0644**.
- The `key_id` is a ULID assigned at generation time.  It appears in every receipt's
  `signature.key_id` field.

### 3.2 Key Resolution (verify)

When verifying receipts, the `key_id` in the receipt chain is used to locate the public
key.  Resolution order:

1. Permanent local cache: `~/.nonsudo/key-cache/<key_id>.jwk`
2. Proxy-written local key: `~/.nonsudo/keys/<key_id>.jwk`
3. Remote fetch: `https://schemas.nonsudo.com/.well-known/keys/<key_id>.json`
   (cached on success in the permanent cache)

The `key_id` is validated against the pattern `/^[a-zA-Z0-9_-]{1,64}$/` before any file
system or network access to prevent path-traversal attacks.  The `key_url` field does not
exist on receipts; key resolution is never driven by receipt content.

### 3.3 Key Rotation

Key rotation is not yet automated.  To rotate, generate a new key pair (via `nonsudo init`
or `nonsudo keys`), update the proxy configuration, and re-publish the new public JWK.
Existing receipts remain verifiable with the old key.

## 4. Verification Layers

Verification is structured in three independent layers (L1, L2, L3).

### 4.1 L1 — Signature Verification

Every receipt carries a `signature` block containing:

```
{
  "alg": "EdDSA",
  "key_id": "<key_id>",
  "sig": "<base64url Ed25519 signature>"
}
```

L1 verification checks that the `sig` field is a valid Ed25519 signature over the
canonical JSON encoding of the receipt (all fields except `signature`), using the
public key identified by `key_id`.

**L1 fails** if any receipt in the chain has an invalid signature.

### 4.2 L2 — Chain Integrity

L2 checks the structural integrity of the receipt chain, independent of key material
(after filtering receipts that already failed L1).

Checks performed:

| Code | Condition |
|------|-----------|
| `MISSING_MANIFEST` | The first receipt is not a `workflow_manifest` |
| `WORKFLOW_ID_MISMATCH` | Any receipt has a `workflow_id` that differs from the first receipt |
| `SEQUENCE_ERROR` | Receipt sequence numbers are not consecutive starting at 0 |
| `INCOMPLETE_CHAIN` | The chain is missing one or more receipts (gap in hash chain) |
| `HASH_MISMATCH` | A receipt's `prev_receipt_hash` does not match the SHA-256 of the preceding receipt |
| `NULL_HASH_EXPECTED` | The first receipt has a non-null `prev_receipt_hash` |

**L2 fails** if any of the above checks produce an error.  L2 errors are attributed to
specific receipt indices for display.

The `--require-complete` flag on `nonsudo verify` additionally requires the last receipt
to be a `workflow_closed` record; if not, the exit code is 1.

### 4.3 L3 — RFC 3161 Timestamping (optional)

L3 is optional and operates on a per-receipt sidecar file (`.tsa` NDJSON alongside the
receipts file).  Each sidecar entry is a `TsaRecord` containing a base64-encoded RFC 3161
`TimeStampResp` DER token.

L3 verification:

1. The TSA's `tsa_id` must be in the operator's `accepting_tsa_ids` allowlist.
2. The `TimeStampResp` DER is parsed and the `PKIStatus` must be `granted (0)`.
3. The `timeStampToken` is extracted and the `TSTInfo` decoded.
4. The `messageImprint.hashAlgorithm` must be `SHA-256`.
5. The `messageImprint.hashedMessage` must equal `SHA-256(canonical_receipt_json)`.
6. The `genTime` in `TSTInfo` must be after the receipt's `issued_at`.

**L3 PASS** — all TSA tokens in the chain validate.
**L3 FAIL** — at least one token is invalid.
**L3 SKIPPED** — no sidecar is present (backward-compatible; does not affect the overall
pass/fail unless `--require-timestamps` is set).

Exit codes from `nonsudo verify`:
- `0` — L1 + L2 pass (L3 PASS or SKIPPED)
- `1` — L1 or L2 fails, or `--require-complete` and chain is open
- `2` — L1 + L2 pass but L3 FAIL, or `--require-timestamps` is set and L3 is SKIPPED

## 5. Threat Model

### 5.1 Threats the system addresses

| Threat | Mitigation |
|--------|------------|
| **Receipt tampering** | L1 Ed25519 signature; any field modification invalidates the signature |
| **Receipt insertion** | L2 hash chain; an inserted receipt breaks `prev_receipt_hash` linkage |
| **Receipt deletion / reordering** | L2 sequence numbers + hash chain |
| **Missing chain start** | L2 `MISSING_MANIFEST` check |
| **Cross-chain replay** | L2 `WORKFLOW_ID_MISMATCH` check; all receipts must share one `workflow_id` |
| **Backdated receipts** | L3 RFC 3161 timestamps (when enabled) bind receipts to real wall-clock time |
| **Key substitution** | `key_id` is resolved from the verifier's local trust store, never from receipt content |
| **Path traversal via key_id** | `key_id` validated against `/^[a-zA-Z0-9_-]{1,64}$/` before file/network access |
| **Prompt injection via receipt data** | `nonsudo explain` wraps receipt data in `<receipt_data>` delimiters with an explicit instruction to treat it as data only |
| **Stale conform test vectors** | `nonsudo conform --offline` warns when cached vectors are >90 days old |
| **Undeclared tool use** | Proxy rejects tool calls not listed in `declared_tools` (workflow_manifest) |
| **Policy bypass at step-up** | `STEP_UP` decisions are treated as `BLOCK` at the proxy |

### 5.2 Out of scope / Limitations

- **Private key compromise**: If the signing key is leaked, an attacker can forge receipts.
  The system does not implement key revocation; operators should rotate keys promptly on
  suspected compromise.
- **Proxy process compromise**: The proxy signs in-process.  A compromised proxy process
  can produce valid but false receipts.  Defence-in-depth requires OS-level isolation.
- **Clock skew**: L3 only checks that `genTime ≥ issued_at`.  Receipts with incorrect
  system clocks may pass L3 even if timestamped before the action occurred.
- **TSA trust**: L3 trust is rooted in the operator's `accepting_tsa_ids` list.  Operators
  are responsible for vetting and maintaining this list.
- **Policy completeness**: The system verifies that the *recorded* policy decision was
  correct; it cannot detect decisions that were never recorded (e.g., a proxy crash between
  action and write).  The write serialization queue (`writeQueue`) mitigates races but does
  not eliminate them in the event of hard process termination.

## 6. Conformance Testing

The `nonsudo conform` command verifies that the local verifier implementation correctly
handles the published VAR v1 test-vector suite at
`https://schemas.nonsudo.com/var/v1/test-vectors.json`.

Test vectors cover:
- TV-01..TV-05: Valid chains (expected L1=PASS, L2=PASS)
- TV-06: Tampered signature (expected L1=FAIL)
- TV-07..TV-12: Structural chain failures (expected L2=FAIL)
- TV-13: Valid chain with allowlisted TSA (expected L3=PASS)
- TV-14: Valid chain with non-allowlisted TSA (expected L3=FAIL)
- TV-15: Chain starting with `action_receipt` (expected L2=FAIL, MISSING_MANIFEST)

A vector "passes" when the verifier's actual result matches the declared expectation.
An intentionally-bad vector passes the conformance suite when the verifier correctly
detects the fault.

## 7. Audit Trail Structure

A complete session produces:

```
workflow_manifest  (seq 0, prev_hash=null)
action_receipt     (seq 1, prev_hash=H(manifest))
action_receipt     (seq 2, prev_hash=H(receipt1))
...
workflow_closed    (seq N, prev_hash=H(receiptN-1))
```

Optional sidecar (`.tsa` NDJSON) maps each `receipt_id` to its RFC 3161 token.

The `workflow_manifest` commits to:
- `policy_bundle_hash` — SHA-256 of the policy file evaluated during the session
- `declared_tools` — tools the agent declared it would use
- `session_budget` — resource limits (e.g., max API calls)
- `initiator_id`, `workflow_owner` — identity metadata

These commitments prevent undetected policy swaps or tool injection after the session
starts.
